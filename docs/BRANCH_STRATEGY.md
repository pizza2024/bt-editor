# 分支管理策略

## 分支结构

```
main  ──────────────────────────────────────── 主线分支（生产环境）
  │                                              合并需要 pizza 同意
  │                                            
dev  ─────────────────────── 开发分支            我（bt-pipi）的工作分支
        ↑ 所有功能/修复在此开发
        ↓ 完成后发起 PR 合并到 main
```

## 分支说明

| 分支 | 用途 | 负责人 | 合并规则 |
|------|------|--------|---------|
| `main` | 主线，生产环境代码 | pizza | PR 需要 pizza 审核同意 |
| `dev` | 开发分支，所有功能/修复 | bt-pipi | 直接提交 |

## 工作流程

### 1. 开发新功能或修复
```bash
# 确保在 dev 分支
git checkout dev

# 开始开发...
git add .
git commit -m "feat: 新功能"
git push origin dev
```

### 2. 合并到 main
```bash
# 在 dev 分支创建 PR
git checkout -b feature/xxx  # 可选：从 dev 创建功能分支
# ... 开发完成后 ...
# 通过 GitHub 创建 Pull Request 到 main
```

pizza 审核 PR 后决定是否合并。

## 规则

1. **main 不可直接提交** — 所有变更必须通过 PR
2. **dev 是工作分支** — 我可以直接提交和推送
3. **测试通过后再合并** — 自测是强制要求
4. **PR 需要描述** — 说明做了什么，改了什么

## 当前状态

- `dev` — 开发分支（当前工作分支）
